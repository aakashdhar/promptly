## P1 — Warnings

### P1-001 · DESIGN_TASKS.md · All tasks — No acceptance criteria
**Issue:** All 5 tasks are names/descriptions only. No pass/fail criteria in any task entry. An agent or developer executing these tasks has no way to verify completion beyond "I did the thing."
**Why:** Without acceptance criteria, verification is convention not evidence. Exactly the O'Reilly vague-spec gap.
**Fix:** Add a "Done when:" block to each task with 3–5 specific, visually-verifiable criteria.
**Example for TOG-001:**
```
Done when:
✅ Expand button visible at top-right of 28px drag row
✅ Button is 22×22px, border-radius 6px, WebkitAppRegion no-drag
✅ IdleState height div reads 134px
✅ "built using vibe-*" tagline at bottom: 10px
✅ Content row (mic ring, text, type btn, mode pill) pixel-identical
```

### P1-002 · DESIGN_TASKS.md / DESIGN_PLAN.md · TOG-004 — Non-actionable task
**Issue:** TOG-004 "verify window resize fires correctly" says in its own description "(transition() already calls resizeWindow — no extra code needed)." No code to write, no file to change, no pass/fail criterion. It is a spec comment masquerading as a task.
**Why:** An agent hitting this task will either skip it or waste time confirming nothing changed. Verification logic belongs inside TOG-005's smoke checklist, not as a standalone task slot.
**Fix:** Remove TOG-004. Absorb window-resize verification into TOG-005: "After collapse button click, window height = STATE_HEIGHTS.IDLE (134px). After expand button click, window height = STATE_HEIGHTS.PROMPT_READY (560px)."

### P1-003 · DESIGN_SPEC.md · onClick behaviour — Empty-state UX undefined
**Issue:** Expand button onClick says "Call transition(STATES.PROMPT_READY) — pure state change, no new logic." No UX defined for the case where `generatedPrompt` is empty (first launch, or after Reset). PROMPT_READY renders "✓ Prompt ready" + empty YOU SAID + active Copy button with empty content — misleading affordance.
**Why:** An agent following this spec verbatim ships a confusing empty state. The code review (P2-TOG-002) caught and fixed this reactively. The spec should have required it proactively.
**Fix:** Add to expand button section:
> "If `generatedPrompt` is empty: expand button visible but dimmed (opacity 0.35), cursor default, non-interactive. Transitions to PROMPT_READY only when generatedPrompt is non-empty."

### P1-004 · DESIGN_SPEC.md · IMPORTANT CONSTRAINTS — "Zero logic changes" too absolute
**Issue:** "Zero logic changes" is stated as a hard constraint. It was violated by the necessary post-review fix: `onExpand={generatedPrompt ? () => transition(...) : null}`. The constraint caused the spec to omit the empty-state case entirely (P1-003), and the implementation had to deviate from the spec to be correct.
**Why:** Absolute constraints that turn out to be wrong create worse outcomes than qualified constraints. An agent following "zero logic changes" verbatim either ships the broken empty state or introduces uncertainty by deviating without spec justification.
**Fix:** Replace "Zero logic changes" with:
> "No new business logic — no new IPC, no new hooks, no new utility functions. Minimal conditional display logic based on existing React state is acceptable (e.g. show/hide button, dim/enable based on generatedPrompt)."

## Verdict

**⚠️ SPEC HAS WARNINGS — 0 P0, 4 P1 findings**

The spec provided enough direction to build the correct feature. The critical gap is acceptance criteria (P1-001) — 5 tasks shipped without formal pass/fail definitions. P1-003 and P1-004 reflect a constraint that was too absolute and an empty-state case the spec didn't anticipate — both were caught reactively by the build review.

**Primary lesson for future design specs:** Every task needs a "Done when:" block. Constraints like "zero logic changes" should be qualified ("no business logic"), not absolute.
